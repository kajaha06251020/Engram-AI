# Engram-AI v0.1 Design Specification

## Overview

Engram-AI is an experience-driven memory infrastructure for AI agents. Unlike text-based memory systems (e.g., Mem0) that store facts like a "diary," Engram-AI records causal structures — action, context, outcome — like "scars." Agents learn from what they did and what happened, not just what they read.

**Tagline:** "Today's AI memory is a diary. Engram-AI is a scar."

## Decisions

| Item | Decision |
|------|----------|
| Form | Python library (OSS) |
| Target Users | AI agent developers (LangChain/CrewAI/AutoGen ecosystem) |
| v0.1 Scope | record + query + crystallize + evolve (full set) |
| Storage | ChromaDB (vector search, embedded in pip install) |

| Repository | `F:/playground/Engram-AI` |

## 1. Core Concept

### The Difference Between "Diary" and "Scar"

A child who reads "stoves are hot" in a textbook vs. a child who touches a stove and gets burned. Which one never touches the stove again?

Current AI memory stores text (semantic memory). Engram-AI stores experience (procedural memory) — the causal structure of "I did X in situation Y and Z happened." This is fundamentally a different data structure.

### The Four Operations

1. **record** — Record an experience (action → context → outcome)
2. **query** — Find the best action for a given situation based on past experiences
3. **crystallize** — Extract patterns from accumulated experiences into reusable skills
4. **evolve** — Persist learned skills into agent configuration (CLAUDE.md)

## 2. Data Models

All models use Pydantic `BaseModel` for validation, serialization, and automatic JSON schema generation (required for MCP tool definitions).

### Experience

```python
class Experience(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid4()))
    schema_version: int = 1                    # For future migrations
    action: str                                # What was done
    context: str                               # Situation description
    outcome: str                               # What happened as a result
    valence: float = Field(ge=-1.0, le=1.0)    # Good (+1.0) to bad (-1.0)
    timestamp: datetime = Field(default_factory=datetime.now)
    metadata: dict = Field(default_factory=dict)  # tool, file, etc.
    status: str = "complete"                   # "pending" | "complete"
```

### Skill

```python
class Skill(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid4()))
    schema_version: int = 1
    rule: str                                  # "Avoid Optional in API responses"
    context_pattern: str                       # When to apply
    confidence: float = Field(ge=0.0, le=1.0)
    source_experiences: list[str]              # Experience IDs
    evidence_count: int
    valence_summary: dict                      # {"positive": 3, "negative": 2}
    created_at: datetime = Field(default_factory=datetime.now)
```

### EvolutionRecord

```python
class EvolutionRecord(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid4()))
    schema_version: int = 1
    timestamp: datetime = Field(default_factory=datetime.now)
    skills_applied: list[str]                  # Skill IDs written to config
    config_path: str                           # Target file
    diff: str                                  # What changed
```

**Key design points:**
- `valence` is what separates "diary" from "scar." Not just facts, but whether the outcome was good or bad, as a number.
- `schema_version` on all models enables future data migrations without breaking existing stored data.
- `status: "pending"` on Experience enables the two-phase recording flow (see Section 5.1).

```
│   ├── skill.py                 ← Skill

│
├── storage/                     ← Storage abstraction layer

│   └── chromadb.py              ← ChromaDB implementation (v0.1)
│
├── llm/                         ← LLM provider abstraction layer
│   ├── base.py                  ← BaseLLM (abstract class)

│
├── adapters/                    ← External framework integration layer
│   ├── base.py                  ← BaseAdapter (abstract class)
│   └── claude_code.py           ← Claude Code / CLAUDE.md integration (v0.1)
│
├── events/                      ← Event system layer
│   ├── bus.py                   ← Event bus (publish/subscribe)
│   └── events.py                ← Event definitions and payloads
│
└── exceptions.py                ← Exception hierarchy
```

**v0.1 scope note:** The following directories are reserved for future expansion and will contain only empty `__init__.py` placeholders in v0.1:
- `core/analyzer.py` — Pattern analysis (features #3, #15, #16)
- `models/emotion.py` — Emotion tagging (feature #1)
- `policies/` — Decay, conflict detection, sandbox (features #4, #6, #10, #14)
- `transfer/` — Export, import, marketplace (features #5, #17, #20)

### Design Principles

1. **Abstract base + Implementation pairs** — Every layer has a `base.py` abstract class. Adding Qdrant, OpenAI, or LangChain means adding a new implementation file without touching existing code.

2. **EventBus for loose coupling** — All cross-cutting concerns (webhooks, decay, conflict detection) subscribe to events. Core logic never knows about them.

3. **Models are independent** — `models/` depends on nothing. Every other layer depends on `models/`. This keeps data structures stable.

4. **forge.py is a thin facade** — Users only interact with `Forge()`. Internal complexity is hidden.

## 4. Claude Code Integration

Engram-AI operates within Claude Code through three mechanisms:

### 4.1 MCP Server

Registered in `~/.claude/settings.json`. Uses the `mcp` Python SDK (from Anthropic):

```json
{
  "mcpServers": {
    "engram-ai": {
      "command": "engram-ai",
      "args": ["serve"]
    }
  }
}
```

Provides tools to Claude Code:
- `engram_record` — Record an experience
- `engram_query` — Search past experiences for best action
- `engram_crystallize` — Crystallize skills from experiences
- `engram_evolve` — Write skills to CLAUDE.md
- `engram_status` — Show statistics

### 4.2 Hooks (Automatic Recording)

Claude Code hook events used: `PostToolUse` and `UserPromptSubmit`.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "engram-ai hook post-tool-use"
          }
        ]
      }
    ],
    "UserPromptSubmit": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "engram-ai hook user-prompt-submit"
          }
        ]
      }
    ]
  }
}
```

**Two-phase recording flow:**

1. `PostToolUse` fires → Engram-AI creates a **pending experience** (action + context, no outcome yet). The hook receives `tool_name`, `tool_input`, and `tool_result` via stdin JSON.
2. `UserPromptSubmit` fires → Engram-AI reads the user's message, detects valence, and **completes the most recent pending experience** with the outcome. If the user's message is unrelated to the pending experience (topic change), the pending experience is completed with `valence: +0.3` (implicit approval).

Hooks are stateless fire-and-forget commands. Correlation between tool use and user response is managed through a local pending experience queue stored in `~/.engram-ai/pending.json`.

### 4.3 CLI Commands

```bash
engram-ai setup          # Auto-configure MCP + Hooks
engram-ai serve          # Start MCP server
engram-ai status         # Show statistics
engram-ai crystallize    # Crystallize skills
engram-ai evolve         # Write skills to CLAUDE.md
engram-ai query "..."    # Search experiences
```

**Note on `/engram` slash commands:** These are implemented as MCP tools, not Claude Code custom slash commands. Users invoke them through natural language ("crystallize my experiences") or by asking Claude Code to use the MCP tool directly.

### 4.4 User Experience Timeline

```
Day 1:    pip install engram-ai && engram-ai setup
          → "Setup complete. Claude Code will now record experiences."

Day 1-7:  Use Claude Code normally. Hooks record experiences silently.

Day 7:    "Crystallize my experiences"
          → Claude Code calls engram_crystallize MCP tool
          → "4 skills crystallized from 132 experiences"
          "Evolve my CLAUDE.md"
          → Claude Code calls engram_evolve MCP tool
          → "4 skills written to CLAUDE.md"

Day 30:   User notices: "Correction requests have dropped dramatically."
```

## 5. Data Flow

### 5.1 record: Experience Recording

```
Phase 1: PostToolUse hook fires
  → Hook receives stdin JSON: {tool_name, tool_input, tool_result}
  → Context extraction:
      - file path + tool name as base context
      - tool_input/tool_result summarized into action description
      (No LLM call — uses structured data from hook stdin)
  → Create pending Experience:
      action: "Edited src/api.py: changed Optional[str] to str"
      context: "Edit tool on src/api/response.py"
      outcome: ""  (empty — waiting for user response)
      status: "pending"
  → Write to ~/.engram-ai/pending.json (append-only log)

Phase 2: UserPromptSubmit hook fires
  → Read user's message from stdin
  → Valence detection (tiered strategy for cost control):
      Tier 1 — Keyword heuristic (FREE, instant):
        "完璧|perfect|great|thanks|良い" → +1.0
        "違う|wrong|fix|no|ダメ"         → -0.8
        "まあ|ok|okay"                   → +0.3
      Tier 2 — If ambiguous, use Claude API (cost: ~0.001$/call):
        Send user message → get valence float
      Tier 3 — If no pending experience or topic changed:
        Complete pending with valence +0.3 (implicit approval)
  → Complete pending Experience with outcome + valence
  → Store in ChromaDB: text → embedding → vector index
  → EventBus.emit("experience.recorded", experience)
```

### 5.2 query: Experience Search

```
Input: context string
  → ChromaDB vector similarity search (cosine similarity)
  → Two-step scoring:
      Step 1: Rank by similarity (relevance to current context)
      Step 2: Partition by valence sign:
        valence > 0  → "best" list (sorted by similarity × valence)
        valence < 0  → "avoid" list (sorted by similarity × abs(valence))
  → Returns:
      best:  [high relevance, positive outcome]  — "Do this"
      avoid: [high relevance, negative outcome]   — "Don't do this"
```

### 5.3 crystallize: Skill Crystallization

```
Step 1: Retrieve all experiences from ChromaDB
Step 2: Cluster by context similarity
    - For each experience, query ChromaDB for similar experiences
      (similarity threshold >= 0.8)
    - Group overlapping results into clusters
    - This leverages ChromaDB's existing similarity search
      (no separate clustering library needed)
Step 3: Filter clusters with >= min_experiences (default: 3)
Step 4: For each cluster, send to Claude API:
    Prompt: "Analyze these experiences and extract a reusable rule.
             Experiences: [cluster data]
             Output: {rule, context_pattern, confidence}"
Step 5: Store each Skill in ChromaDB
Step 6: EventBus.emit("skill.crystallized", skill) for each
```

### 5.4 evolve: Agent Evolution

```
Step 1: Read current CLAUDE.md
Step 2: Get unapplied skills (since last evolve)
Step 3: Claude API generates natural integration text
Step 4: Write to CLAUDE.md using managed section delimiters:

    <!-- engram-ai:start -->
    ## Engram-AI: Learned Skills
    <!-- Auto-generated. Last evolved: 2026-03-17T01:30:00Z -->
    - Avoid Optional in API responses, use default values (confidence: 0.85)
    - Include specific fix suggestions in error messages (confidence: 0.78)
    <!-- engram-ai:end -->

    Write strategy:
    - If delimiters exist: replace content between them
    - If delimiters missing: append section at end of file
    - Always read-before-write, preserve everything outside managed section

Step 5: Record EvolutionRecord
Step 6: EventBus.emit("agent.evolved", evolution_record)
```

## 6. Event System

### Event Catalog (v0.1)

| Event Name | Payload | Emitted By |
|-----------|---------|------------|
| `experience.recorded` | `Experience` (complete) | `Recorder` |
| `experience.pending` | `Experience` (status=pending) | `Recorder` |
| `skill.crystallized` | `Skill` | `Crystallizer` |
| `agent.evolved` | `EvolutionRecord` | `Evolver` |

### Semantics

- **Synchronous execution** in v0.1. Subscribers execute in registration order.
- **Observe-only** — subscribers receive a copy of the payload and cannot modify the original.
- **Fail-open** — if a subscriber raises an exception, it is logged and the next subscriber runs. Core operations are never blocked by subscriber failures.
- **Future:** Async execution and subscriber priority ordering can be added without changing the interface.

### Interface

```python
class EventBus:
    def on(self, event_name: str, callback: Callable) -> None: ...
    def emit(self, event_name: str, payload: BaseModel) -> None: ...
    def off(self, event_name: str, callback: Callable) -> None: ...
```

## 7. Concurrency Model

Hooks fire on every tool use. In a fast Claude Code session, multiple hooks may fire in quick succession.

### Strategy: Append-Only Log + MCP Server as Single Writer

```
Hooks (multiple, concurrent):
  → Write to ~/.engram-ai/pending.json (append-only, file-locked)
  → Each write is a single JSON line (JSONL format)
  → File lock via fcntl/msvcrt for cross-process safety

MCP Server (single process, long-running):
  → Owns ChromaDB connection exclusively
  → Periodically reads pending.json, imports into ChromaDB
  → All query/crystallize/evolve operations go through MCP server
  → No concurrent ChromaDB writes
```

This ensures ChromaDB embedded mode (SQLite-backed) never faces concurrent write issues.

## 8. Error Handling

### Exception Hierarchy

```python
class EngramError(Exception):
    """Base exception for all Engram-AI errors"""

class StorageError(EngramError):
    """ChromaDB connection failure, corruption, etc."""

class LLMError(EngramError):
    """Claude API unreachable, rate limited, etc."""

class CrystallizationError(EngramError):
    """Not enough experiences, no patterns found, etc."""

class EvolutionError(EngramError):
    """Config file read-only, write failure, etc."""

class HookError(EngramError):
    """Hook stdin parse failure, pending queue error, etc."""
```

### Error Behavior

| Scenario | Behavior |
|----------|----------|
| ChromaDB corrupted | `StorageError` — log, suggest `engram-ai repair` |
| Claude API unreachable | `LLMError` — crystallize/evolve fail gracefully, record still works (no LLM needed for recording) |
| CLAUDE.md read-only | `EvolutionError` — report file permissions issue |
| Hook stdin parse failure | `HookError` — log silently, skip this event (never block user's Claude Code session) |
| Pending queue file locked | Retry with backoff (3 attempts), then log and skip |

**Critical principle:** Hooks must NEVER block or slow down the user's Claude Code experience. All hook errors fail silently with logging.

## 9. Test Strategy

### Three-Layer Testing

| Layer | Scope | LLM | Speed |
|-------|-------|-----|-------|
| Unit | Each module independently | Mock | Fast |
| Integration | record→crystallize→evolve flow | Mock | Medium |
| E2E | Claude Code actual usage | Real Claude API | Slow (main branch only) |

### Key Test Cases

**Unit:**
- Experience creation and Pydantic validation
- Valence range validation (-1.0 to +1.0)
- Query scoring: best/avoid partitioning correctness
- Skill generation with required fields
- CLAUDE.md delimiter-based write (create, update, preserve)
- Event subscription, emission, and fail-open behavior
- Keyword-based valence detection accuracy
- Pending experience queue JSONL read/write

**Integration:**
- Full flow: 5 experiences → crystallize → at least 1 skill
- Full flow: record → crystallize → evolve → CLAUDE.md updated
- Idempotent evolve (no duplicates on re-run)
- Event chain fires completely
- Two-phase recording: pending → complete
- Concurrent hook writes to pending.json

**E2E:**
- MCP server starts successfully
- Hook captures PostToolUse events
- `engram-ai setup` configures settings.json correctly
- CLI commands return expected output
- Full Day 1-7 simulation
