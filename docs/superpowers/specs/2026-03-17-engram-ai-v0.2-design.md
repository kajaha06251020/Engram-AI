# Engram-AI v0.2 Design Specification

## Overview

v0.2 adds six features to the existing v0.1 foundation, organized around three themes: **learning quality** (Skill Reinforcement, Confidence Decay), **data richness** (Experience Chain, Anti-Skill), **safety & reach** (Skill Conflict Detection, Framework Adapters).

**Release:** Single v0.2 release containing all six features.

**Implementation approach:** Layer-ordered (models → policies → core extensions → adapters) to minimize rework and keep dependencies clean.

---

## Decisions

| Item | Decision |
|------|----------|
| Release strategy | Single v0.2 release |
| Decay model | Time-based (90-day half-life) + event-based (contradiction penalty) |
| Reinforcement trigger | At crystallize time only (pattern-level, not per-experience) |
| Experience linking | Automatic (semantic similarity) + explicit (parent_id) |
| Conflict resolution | Notification by default, optional auto-merge via CLI/MCP |
| Anti-Skill output | Separate section with dedicated delimiters |
| Framework targets | Cursor (.cursorrules), Gemini CLI (GEMINI.md), Windsurf (.windsurfrules) |
| Backward compatibility | schema_version 1 → 2, all new fields Optional or have defaults; schema_version is informational only and never branched on |
| Breaking changes | Adding `write_anti_skills` to `BaseAdapter` and `verify_conflict`/`merge_skills` to `BaseLLM` are known breaking changes for custom v0.1 subclasses. Mitigated by providing concrete defaults that raise `NotImplementedError`. |

## 1. Data Model Changes

### 1.1 Experience Model

```python
class Experience(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid4()))
    schema_version: int = 2
    action: str
    context: str
    outcome: str
    valence: float = Field(ge=-1.0, le=1.0)
    timestamp: datetime = Field(default_factory=datetime.now)
    metadata: dict = Field(default_factory=dict)
    status: str = "complete"

    # v0.2: Experience Chain
    parent_id: str | None = None
    related_ids: list[str] = Field(default_factory=list)
```

- `parent_id`: Explicit link to the immediately preceding experience (set by hooks or manual recording).
- `related_ids`: Automatically populated via semantic similarity search at record time (up to 5, similarity >= 0.5).
- Both fields are Optional, so v0.1 data deserializes without error.

### 1.2 Skill Model

```python
class Skill(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid4()))
    schema_version: int = 2
    rule: str
    context_pattern: str
    confidence: float = Field(ge=0.0, le=1.0)
    source_experiences: list[str]
    evidence_count: int
    valence_summary: dict
    created_at: datetime = Field(default_factory=datetime.now)

    # v0.2: Reinforcement & Decay
    last_reinforced_at: datetime | None = None
    reinforcement_count: int = 0

    # v0.2: Anti-Skill
    skill_type: str = "positive"  # "positive" | "anti"

    # v0.2: Conflict Detection
    conflicts_with: list[str] = Field(default_factory=list)

    # v0.2: Lifecycle status
    status: str = "active"  # "active" | "superseded"
```

- `skill_type` uses a string rather than an enum for forward compatibility.
- `conflicts_with` stores mutual references (both skills in a conflict pair reference each other).
- `status` tracks lifecycle: `"active"` skills are in use, `"superseded"` skills have been replaced by a merged skill. `get_all_skills()` and `get_unapplied_skills()` filter to `status == "active"` only.

### 1.3 New Events

```python
# Added to events/events.py
SKILL_REINFORCED = "skill.reinforced"
SKILL_CONFLICT_DETECTED = "skill.conflict.detected"
SKILL_DECAYED = "skill.decayed"
EXPERIENCE_CHAINED = "experience.chained"
```

Event payloads follow the same pattern as v0.1:
- `SKILL_REINFORCED`: the updated `Skill` object
- `SKILL_CONFLICT_DETECTED`: a `dict` with `{"skill_a": Skill, "skill_b": Skill}`
- `SKILL_DECAYED`: the updated `Skill` object
- `EXPERIENCE_CHAINED`: the `Experience` object (same as `EXPERIENCE_RECORDED`)

### 1.4 Schema Version & Backward Compatibility

`schema_version` is **purely informational** — no code path branches on its value. v0.1 data stored in ChromaDB with `schema_version=1` will deserialize correctly into v0.2 model classes because all new fields have defaults. The `schema_version` field's default value is `2` at the class level, but objects deserialized from v0.1 storage will retain `schema_version=1`. No lazy upgrade or normalization is performed.

## 2. Storage Extensions

Three new abstract methods on `BaseStorage`, implemented in `ChromaDBStorage`:

```python
class BaseStorage(ABC):
    # ... existing methods unchanged ...

    @abstractmethod
    def query_skills(self, text: str, k: int = 5) -> list[tuple[Skill, float]]:
        """Semantic search over skills. Used by Crystallizer for reinforcement matching."""

    @abstractmethod
    def update_skill(self, skill: Skill) -> None:
        """Update skill metadata in-place. Used by Reinforcement, Decay, and Conflict."""

    @abstractmethod
    def get_experience(self, experience_id: str) -> Experience | None:
        """Fetch a single experience by ID. Used for chain traversal."""
```

### ChromaDBStorage Implementation

- `query_skills`: Mirrors `query_experiences` — queries the `skills` collection by text similarity, returns `(Skill, similarity)` tuples. Filters to `status == "active"` only.
- `update_skill`: Rebuilds the full metadata dict from the `Skill` object, including all flat fields (`confidence`, `applied`, `status`) in addition to the `data` JSON blob. This ensures flat metadata fields used by ChromaDB `where` filters stay in sync with the serialized object.
- `get_experience`: Calls `self._experiences.get(ids=[id])`, returns `None` if not found.

### update_skill Implementation Detail

```python
def update_skill(self, skill: Skill) -> None:
    doc = f"{skill.rule} | {skill.context_pattern}"
    existing = self._skills.get(ids=[skill.id])
    applied = existing["metadatas"][0].get("applied", "false") if existing["metadatas"] else "false"
    self._skills.update(
        ids=[skill.id],
        documents=[doc],
        metadatas=[{
            "data": skill.model_dump_json(),
            "confidence": skill.confidence,
            "applied": applied,
            "status": skill.status,
        }],
    )
```

Critical: `update_skill` must overwrite **all** flat metadata keys (`confidence`, `applied`, `status`) in tandem with the `data` JSON blob. The `applied` flag is preserved from the existing record unless the `Skill` object explicitly changes it.

### get_all_skills / get_unapplied_skills Filter Update

Both methods add `where={"status": "active"}` to exclude superseded skills:

- `get_all_skills()`: `self._skills.get(where={"status": "active"})`
- `get_unapplied_skills()`: `self._skills.get(where={"$and": [{"applied": "false"}, {"status": "active"}]})`

Note: v0.1 stored skills lack the `status` metadata field. On first v0.2 access, a migration adds `"status": "active"` to existing skill records via a `_migrate_v1_skills()` method called in `ChromaDBStorage.__init__`. The migration is **idempotent** — it uses a ChromaDB `where` filter to update only records missing the `status` field (e.g., `where={"status": {"$exists": false}}` or equivalent), so repeated calls at startup have no effect on already-migrated records and incur no O(N) rewrite cost.

## 3. Policies Layer

New `policies/` package with two EventBus subscribers. Both are pluggable — they register via `event_bus.on()` and operate through storage/LLM interfaces, requiring no changes to existing core code.

### 3.1 Confidence Decay (`policies/decay.py`)

```python
class DecayConfig(BaseModel):
    half_life_days: float = 90.0
    min_confidence: float = 0.1
    event_penalty: float = 0.15
    event_valence_threshold: float = -0.3  # trigger event decay when valence < this
    event_similarity_threshold: float = 0.3  # skill similarity threshold for event decay

class DecayPolicy:
    def __init__(self, storage, event_bus, config=None):
        self._storage = storage
        self._event_bus = event_bus
        self._config = config or DecayConfig()
        event_bus.on(EXPERIENCE_RECORDED, self._on_experience_recorded)
```

**Time decay** (`apply_time_decay()`):
- Formula: `new_conf = conf × 0.5^(days_elapsed / half_life_days)`
- `days_elapsed` measured from `last_reinforced_at` (if set) or `created_at`.
- Skills dropping below `min_confidence` (0.1) emit `SKILL_DECAYED`.
- Called explicitly via CLI (`engram-ai decay`) or before crystallize. Not a background thread.

**Event decay** (`_on_experience_recorded(experience)`):
- Triggered when a new experience is recorded.
- **Gate:** Only fires if `experience.valence < config.event_valence_threshold` (default: -0.3). Mildly negative experiences do not trigger decay.
- Searches for semantically similar skills via `storage.query_skills(experience.context)`.
- For each matched skill with `similarity >= config.event_similarity_threshold` (default: 0.3): derive the skill's valence direction as `positive_count - negative_count` from `valence_summary` (e.g., `{"positive": 5, "negative": 1}` → direction = +4, meaning positive). If the experience's valence and the skill's direction are opposite signs → subtract `event_penalty` (0.15) from confidence.
- Clamp confidence to `max(0.0, new_conf)`. Emit `SKILL_DECAYED` if confidence drops below `min_confidence`.
- Call `storage.update_skill()` to persist.

### 3.2 Skill Conflict Detection (`policies/conflict.py`)

```python
class ConflictPolicy:
    def __init__(self, storage, event_bus, llm):
        self._storage = storage
        self._event_bus = event_bus
        self._llm = llm
        event_bus.on(SKILL_CRYSTALLIZED, self._on_skill_crystallized)
```

**Auto-detection** (`_on_skill_crystallized(new_skill)`):
1. Semantic search for similar skills via `storage.query_skills(new_skill.rule, k=10)`.
2. Filter candidates: similarity >= 0.3 + opposing valence direction (derived from `valence_summary` as in DecayPolicy).
3. LLM confirmation: call `self._llm.verify_conflict(new_skill, candidate)` to reduce false positives.
4. Confirmed conflicts → update both skills' `conflicts_with` lists via `storage.update_skill()`.
5. Emit `SKILL_CONFLICT_DETECTED` with payload `{"skill_a": new_skill, "skill_b": candidate}`.

**Manual detection** (`detect_all_conflicts() -> list[tuple[Skill, Skill]]`):
- Semantic pre-filter only (no LLM calls) to avoid O(N^2) API cost.
- For each skill, query similar skills and check opposing valence direction.
- Returns candidate pairs for human review. The user can then confirm and optionally merge via `auto_merge`.

**Auto-merge** (`auto_merge(skill_a_id, skill_b_id) -> Skill`):
- Retrieves both skills from storage.
- Calls `self._llm.merge_skills(skill_a, skill_b)` to produce a merged skill.
- Stores the merged skill via `storage.store_skill()`.
- Marks both original skills as `status = "superseded"` via `storage.update_skill()`.
- Returns the merged skill.

### 3.3 Forge Wiring

`Forge.__init__` instantiates both policies after creating the core components:

```python
class Forge:
    def __init__(self, storage=None, llm=None, adapter=None,
                 decay_config=None, enable_policies=True):
        # ... existing init (storage, event_bus, llm, adapter, core components) ...

        # v0.2: Policies (opt-out via enable_policies=False for testing)
        if enable_policies:
            self._decay_policy = DecayPolicy(self._storage, self._event_bus, decay_config)
            self._conflict_policy = ConflictPolicy(self._storage, self._event_bus, self._llm)
        else:
            self._decay_policy = None
            self._conflict_policy = None

    # Expose policy operations
    def apply_decay(self) -> list[Skill]:
        if self._decay_policy:
            return self._decay_policy.apply_time_decay()
        return []

    def detect_conflicts(self) -> list[tuple[Skill, Skill]]:
        if self._conflict_policy:
            return self._conflict_policy.detect_all_conflicts()
        return []

    def merge_skills(self, skill_a_id, skill_b_id) -> Skill:
        if not self._conflict_policy:
            raise EngramError("Conflict policy not enabled")
        return self._conflict_policy.auto_merge(skill_a_id, skill_b_id)
```

- `enable_policies=False` provides a testing seam to disable EventBus side effects.
- `decay_config` allows custom `DecayConfig` injection.
- Policy methods are exposed as thin wrappers on `Forge` for CLI/MCP access.

## 4. Core Logic Extensions

### 4.1 Crystallizer — Skill Reinforcement

Modified flow in `crystallize()`:

1. Cluster experiences (unchanged).
2. LLM generates candidate skill from cluster (unchanged).
3. **New:** Search existing skills via `storage.query_skills(candidate.rule)`.
4. **If match found** (similarity >= 0.5):
   - Do not store the candidate as a new skill.
   - Update the matching skill: `confidence = min(1.0, confidence + 0.1)`, `reinforcement_count += 1`, `last_reinforced_at = now`, extend `source_experiences` with cluster experience IDs.
   - Call `storage.update_skill()` and emit `SKILL_REINFORCED`.
5. **If no match:** Store as new skill (existing behavior).

Note: The reinforcement similarity threshold is 0.5 (not 0.7), aligned with ChromaDB's default embedding model which produces max ~0.59 cosine similarity for near-duplicate short texts. At 0.7, reinforcement would almost never fire.

No changes to `_cluster_experiences()`. No interface changes.

### 4.2 Recorder — Experience Chain

Updated signatures:

```python
def record(self, action, context, outcome, valence,
           metadata=None, parent_id=None) -> Experience:

def record_pending(self, action, context, metadata=None,
                   parent_id=None) -> None:
```

Modified flow in `record()`:

1. Accept new optional `parent_id` parameter.
2. **Auto-link:** Call `storage.query_experiences(context)`, collect IDs with similarity >= 0.5 (max 5) into `related_ids`.
3. Create Experience with `parent_id` and `related_ids` set.
4. Store and emit `EXPERIENCE_RECORDED` (unchanged).
5. **If any links exist** (parent_id or related_ids non-empty), also emit `EXPERIENCE_CHAINED` with the Experience object as payload.

`record_pending(parent_id=...)` stores the parent_id in the pending Experience object. `complete_pending()` reads it back from the pending queue and passes it through to `record()`.

`Forge.record()` and `Forge.record_pending()` also accept the new `parent_id` parameter and pass it through.

### 4.3 Crystallizer — Anti-Skill Generation

When the LLM generates a skill from a cluster of predominantly negative-valence experiences:
- The skill's `skill_type` is set to `"anti"` (determined by the LLM based on the cluster's valence distribution).
- Anti-skills follow the same storage and reinforcement paths as positive skills.

## 5. Framework Adapters

### 5.1 BaseAdapter Extension

```python
class BaseAdapter(ABC):
    @abstractmethod
    def write_skills(self, config_path: str, skills_text: str) -> None: ...
    @abstractmethod
    def read_config(self, config_path: str) -> str: ...

    # v0.2: Anti-Skill support (concrete default, not abstract)
    def write_anti_skills(self, config_path: str, anti_skills_text: str) -> None:
        raise NotImplementedError(
            f"{type(self).__name__} does not support write_anti_skills. "
            "Upgrade your adapter to v0.2 to use anti-skill features."
        )
```

`write_anti_skills` is a **concrete method** with a `NotImplementedError` default, not an `@abstractmethod`. This preserves backward compatibility: existing v0.1 custom adapter subclasses continue to work. Anti-skill writes will fail explicitly if the adapter hasn't been updated.

### 5.2 ClaudeCodeAdapter — Anti-Skill Section

```python
# Existing
START_DELIMITER = "<!-- engram-ai:start -->"
END_DELIMITER = "<!-- engram-ai:end -->"

# New
ANTI_START = "<!-- engram-ai:anti-skills:start -->"
ANTI_END = "<!-- engram-ai:anti-skills:end -->"
```

`write_anti_skills()` uses the same replace-or-append logic as `write_skills()` with anti-skill delimiters and section title `"## Engram-AI: Anti-Patterns (Avoid)"`.

### 5.3 Shared Delimiter Utility (`adapters/_delimiters.py`)

Extract the replace-or-append logic shared by ClaudeCodeAdapter and GeminiAdapter into a utility function:

```python
def write_delimited_section(path, content, start_delim, end_delim, title, timestamp):
    """Replace or append a delimited section in a file."""
```

This avoids duplicating ~30 lines of non-trivial regex logic between Markdown-based adapters.

CursorAdapter and WindsurfAdapter use a parallel `#`-comment version of this utility:

```python
def write_comment_delimited_section(path, content, start_delim, end_delim, title, timestamp):
    """Replace or append a #-comment-delimited section in a file."""
```

### 5.4 CursorAdapter (`adapters/cursor.py`)

- Target file: `.cursorrules`
- Delimiter format: `# --- engram-ai:start ---` / `# --- engram-ai:end ---`
- Anti-skill delimiters: `# --- engram-ai:anti-skills:start ---` / `# --- engram-ai:anti-skills:end ---`
- Plain text format (no Markdown headings), uses `#` comment-based delimiters.

### 5.5 GeminiAdapter (`adapters/gemini.py`)

- Target file: `GEMINI.md`
- Delimiter format: Same HTML comment style as ClaudeCodeAdapter.
- Uses shared `write_delimited_section` utility.

### 5.6 WindsurfAdapter (`adapters/windsurf.py`)

- Target file: `.windsurfrules`
- Delimiter format: Same `#` comment style as CursorAdapter.
- Uses shared `write_comment_delimited_section` utility.

### 5.7 Evolver — Anti-Skill Routing

```python
def evolve(self, config_path) -> EvolutionRecord | None:
    unapplied = self._storage.get_unapplied_skills()
    if not unapplied:
        return None

    positive = [s for s in unapplied if s.skill_type == "positive"]
    anti = [s for s in unapplied if s.skill_type == "anti"]

    if positive:
        skills_text = self._llm.generate_evolution_text(positive)
        self._adapter.write_skills(config_path, skills_text)
    if anti:
        anti_text = self._llm.generate_evolution_text(anti)
        self._adapter.write_anti_skills(config_path, anti_text)

    # Mark ALL skills (both positive and anti) as applied
    skill_ids = [s.id for s in unapplied]
    self._storage.mark_skills_applied(skill_ids)

    record = EvolutionRecord(
        skills_applied=skill_ids,
        config_path=config_path,
        diff=f"+ {len(positive)} skills, + {len(anti)} anti-skills added",
    )
    self._event_bus.emit(AGENT_EVOLVED, record)
    return record
```

### 5.8 Adapter Registry & CLI Resolution

An adapter registry maps CLI names to classes and default config paths:

```python
# adapters/__init__.py
ADAPTER_REGISTRY = {
    "claude-code": {"class": ClaudeCodeAdapter, "default_config": "CLAUDE.md"},
    "cursor": {"class": CursorAdapter, "default_config": ".cursorrules"},
    "gemini": {"class": GeminiAdapter, "default_config": "GEMINI.md"},
    "windsurf": {"class": WindsurfAdapter, "default_config": ".windsurfrules"},
}
```

`Forge` gains a factory method:

```python
@classmethod
def with_adapter(cls, adapter_name="claude-code", **kwargs) -> "Forge":
    if adapter_name not in ADAPTER_REGISTRY:
        raise EngramError(
            f"Unknown adapter: {adapter_name!r}. "
            f"Valid options: {list(ADAPTER_REGISTRY)}"
        )
    entry = ADAPTER_REGISTRY[adapter_name]
    adapter = entry["class"]()
    return cls(adapter=adapter, **kwargs)
```

CLI's `--adapter` flag uses this registry to resolve the adapter class and default config path. `Forge.evolve()` signature is unchanged; the adapter is set at construction time.

## 6. LLM Interface Extensions

### 6.1 New BaseLLM Methods

```python
class BaseLLM(ABC):
    # Existing (unchanged)
    @abstractmethod
    def detect_valence(self, message: str) -> float: ...
    @abstractmethod
    def crystallize_pattern(self, experiences: list[Experience]) -> Skill | None: ...
    @abstractmethod
    def generate_evolution_text(self, skills: list[Skill]) -> str: ...

    # v0.2: Conflict Detection (concrete defaults, not abstract — preserves v0.1 subclass compat)
    def verify_conflict(self, skill_a: Skill, skill_b: Skill) -> bool:
        """Ask LLM to confirm whether two skills genuinely contradict each other."""
        raise NotImplementedError(
            f"{type(self).__name__} does not support verify_conflict. "
            "Upgrade your LLM implementation to v0.2 to use conflict detection."
        )

    def merge_skills(self, skill_a: Skill, skill_b: Skill) -> Skill:
        """LLM produces a merged skill from two conflicting skills."""
        raise NotImplementedError(
            f"{type(self).__name__} does not support merge_skills. "
            "Upgrade your LLM implementation to v0.2 to use skill merging."
        )
```

### 6.2 ClaudeLLM Implementation

- `verify_conflict`: Sends both skills' rules and context patterns to Claude, asks for a boolean judgment.
- `merge_skills`: Sends both skills to Claude, asks for a unified rule that resolves the contradiction. Returns a new `Skill` with combined `source_experiences` and the higher confidence.

### 6.3 MockLLM Extension

`MockLLM` in `tests/conftest.py` gains:
- `verify_conflict(skill_a, skill_b) -> bool`: Returns `True` if valence_summary directions differ (simple heuristic).
- `merge_skills(skill_a, skill_b) -> Skill`: Returns a new Skill with `rule = f"Merged: {skill_a.rule} + {skill_b.rule}"` and the higher confidence.

### 7.1 CLI

| Command | Description |
|---------|-------------|
| `engram-ai decay` | Apply time-based confidence decay to all skills |
