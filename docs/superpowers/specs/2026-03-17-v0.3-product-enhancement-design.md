# Engram-AI v0.3 Product Enhancement Design

## Overview

Three features to strengthen Engram-AI as a production-ready experience-driven memory system:

1. **Multi-Project Support** — Folder-based project isolation
2. **Scheduler** — Dashboard-embedded background automation
3. **Dashboard Enhancement** — Graph visualization for chains/conflicts, skill decay indicators

## Goals

- Allow users to manage multiple independent projects without data cross-contamination
- Automate decay, conflict detection, and crystallization without manual CLI invocation
- Make v0.2 features (chains, conflicts, decay) visible in the dashboard

## Non-Goals

- Cross-project skill sharing or search
- External scheduler integration (cron, systemd)
- New dashboard pages (enhance existing pages only)

## 1. Multi-Project Support

### Architecture

A `ProjectManager` class acts as a factory that resolves a project name to a `Forge` instance. Forge itself requires one minor fix (pending.jsonl path).

```
ProjectManager
  ├── get_forge("web-app") → Forge(storage_path="~/.engram-ai/data/web-app/")
  ├── get_forge("cli-tool") → Forge(storage_path="~/.engram-ai/data/cli-tool/")
  └── list_projects() → ["web-app", "cli-tool", "default"]
```

### Data Layout

```
~/.engram-ai/
  config.json          # Global config with default_project
  data/
    default/           # Default project (backward-compatible)
      chroma.sqlite3   # ChromaDB data
      pending.jsonl    # Pending experiences (inside project folder)
    web-app/
    cli-tool/
```

### Forge Pending Path Fix

The current `Forge.__init__` calculates pending.jsonl path as `Path(storage_dir).parent / "pending.jsonl"`, which places it in the shared parent directory. This must be fixed to place `pending.jsonl` inside the storage directory to prevent cross-project contamination:

```python
# Before (v0.2):
pending_path = str(Path(storage_dir).parent / "pending.jsonl")

# After (v0.3):
pending_path = str(Path(storage_dir) / "pending.jsonl")
```

This is a minor Forge change. Existing single-project users are unaffected because their pending.jsonl is recreated on each session.

### ProjectManager (`src/engram_ai/project.py`)

```python
class ProjectManager:
    def __init__(self, base_path: Path, llm: BaseLLM, config: dict):
        ...

    def get_forge(self, project: str | None = None) -> Forge:
        """Return Forge for the given project with enable_policies=True.
        Uses default_project from config if project is None.
        Caches Forge instances by project name (dict-based)."""

    def list_projects(self) -> list[str]:
        """List project names (subdirectories of data/)."""

    def delete_project(self, name: str) -> None:
        """Remove a project's data directory. Raises ValueError for 'default'."""

    def get_all_forges(self) -> dict[str, Forge]:
        """Return Forge instances for all existing projects. Used by Scheduler."""
```

- `get_forge()` creates Forge with `enable_policies=True` so scheduler operations work
- Project names are validated: alphanumeric, hyphens, underscores only
- Attempting to delete `"default"` raises an error
- Projects are created implicitly when `get_forge("new-name")` is called for the first time

### Backward Compatibility

Existing data at `~/.engram-ai/data/` (ChromaDB files directly in this directory, not in a subdirectory) is detected on first use. If `data/chroma.sqlite3` exists (ChromaDB's file), ProjectManager treats this path directly as the `"default"` project via path resolution — no symlinks, no data migration required.

On Windows, symlinks require elevated privileges, so direct path resolution is the only approach used.

### CLI Changes

All commands gain a `--project` / `-p` option:

```bash
engram-ai status --project web-app
engram-ai crystallize -p cli-tool
engram-ai query "API design" -p web-app
```

New commands:
```bash
engram-ai projects list          # List all projects
engram-ai projects delete <name> # Delete a project (with confirmation prompt)
```

Omitting `--project` uses `default_project` from config.json (defaults to `"default"`).

### Dashboard Integration

The Dashboard stores a `ProjectManager` on `app.state` instead of a single `Forge`:

```python
# server.py
app.state.project_manager = project_manager

# api.py — modified get_forge dependency
def get_forge(request: Request) -> Forge:
    project = request.query_params.get("project")
    return request.app.state.project_manager.get_forge(project)
```

The WebSocket endpoint is similarly updated to resolve Forge from the `project` query parameter.

All API endpoints accept `?project=<name>` query parameter. The Dashboard UI gets a project selector dropdown in the header. Switching projects reloads all data via API. Default: `"default"` project.

The `useApi` hook in the frontend is updated to accept a `project` parameter that is appended as a query string.

### Config Changes

```json
{
  "default_project": "default"
}
```

## 2. Scheduler (Dashboard-Embedded Automation)

### Architecture

The Scheduler is **global** — it operates on all projects, not just the active dashboard project. This means:
- Project switching in the dashboard does not affect the scheduler
- Decay and conflict detection run across all projects on each tick
- Crystallize counters are tracked per-project

```
Dashboard (FastAPI)
  └── Scheduler (global, uses ProjectManager)
        ├── TimerJob: decay on ALL projects (every 6h)
        ├── TimerJob: conflict detection on ALL projects (every 12h)
        └── CounterJob: crystallize per-project (every 10 experiences)
```

### Scheduler Class (`src/engram_ai/scheduler.py`)

```python
@dataclass
class SchedulerConfig:
    decay_interval_hours: float = 6.0
    conflict_interval_hours: float = 12.0
    crystallize_threshold: int = 10
    enabled: bool = True

class Scheduler:
    def __init__(self, project_manager: ProjectManager, config: SchedulerConfig):
        ...

    async def start(self) -> None:
        """Start background tasks. Called from FastAPI lifespan."""

    async def stop(self) -> None:
        """Cancel all background tasks gracefully."""
```

### Integration with Dashboard

The Dashboard's FastAPI lifespan hook creates and starts the Scheduler:

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    scheduler = Scheduler(app.state.project_manager, scheduler_config)
    app.state.scheduler = scheduler
    await scheduler.start()
    yield
    await scheduler.stop()
```

### Event-Based Crystallize

The Scheduler subscribes to `EXPERIENCE_RECORDED` via EventBus on each project's Forge. Per-project counters track experience counts. When a project's counter reaches `crystallize_threshold`, crystallize runs for that project and its counter resets.

Counters are **ephemeral** (not persisted across dashboard restarts). This is acceptable because:
- Crystallize also runs on timer-based triggers or manual CLI invocation
- At worst, a restart delays crystallization by up to `threshold - 1` experiences

### Timer-Based Jobs

On each tick, the scheduler iterates over `project_manager.get_all_forges()` and calls:
- `forge.apply_decay()` for decay jobs
- `forge.detect_conflicts()` for conflict jobs

Errors in individual project operations are logged and skipped (fail-open). The scheduler continues to the next project and next tick.

### Execution Results

All automated actions emit events through EventBus (SKILL_DECAYED, SKILL_CONFLICT_DETECTED, SKILL_CRYSTALLIZED), which flow through the existing WebSocket to the Dashboard. No new event types needed.

### New Dashboard API

```
GET  /api/scheduler/status    → SchedulerStatus (see types below)
POST /api/scheduler/toggle    → Enable/disable scheduler (global, not per-project)
```

### Config Changes

```json
{
  "scheduler": {
    "decay_interval_hours": 6,
    "conflict_interval_hours": 12,
    "crystallize_threshold": 10,
    "enabled": true
  }
}
```

## 3. Dashboard Enhancement

### 3.1 Graph Page Enhancements

The existing `/graph` page uses react-force-graph-2d. Current edge types: `"source"` (skill→experience) and `"similarity"`.

**New edge types:**

| Edge Type | Meaning | Visual |
|-----------|---------|--------|
| `chain` | Experience parent→child (parent_id) | Blue dashed arrow |
| `related` | Auto-linked experiences (related_ids) | Light gray thin line |
| `conflict` | Conflicting skills (conflicts_with) | Red dashed line |

**New node states:**

| State | Visual |
|-------|--------|
| Superseded skill | Semi-transparent (opacity 0.3) |
| Anti-skill | Red border ring |
| Low confidence (< 0.3) | Pulsing/faded node |

**Graph API changes (`GET /api/graph`):**

Current response:
```json
{
  "nodes": [{"id": "...", "type": "experience|skill", "label": "...", ...}],
  "edges": [{"source": "...", "target": "...", "type": "source|similarity", ...}]
}
```

Enhanced response — same structure, new fields:
```json
{
  "nodes": [
    {
      "id": "...", "type": "skill", "label": "...",
      "confidence": 0.8,
      "skill_type": "positive|anti",
      "status": "active|superseded",
      "conflicts_with": ["skill-id-2"]
    },
    {
      "id": "...", "type": "experience", "label": "...",
      "parent_id": "exp-id-1",
      "related_ids": ["exp-id-2", "exp-id-3"]
    }
  ],
  "edges": [
    {"source": "...", "target": "...", "type": "chain", "weight": 1.0},
    {"source": "...", "target": "...", "type": "conflict", "weight": 0.5}
  ]
}
```

The graph API includes superseded skills (with `status: "superseded"`) so the graph can show the full history.

Note: The v0.2 Pydantic models already serialize these fields via `model_dump()`. The backend API changes are limited to building new edge types from the existing data.

### 3.2 Skills Page Enhancements

Current: table with rule, context, confidence, applied status.

Add:
- **Confidence color bar**: Green (>0.7) → Yellow (0.3–0.7) → Red (<0.3)
- **Status badge**: "active" (green) / "superseded" (gray strikethrough)
- **Type badge**: "positive" (default, no badge) / "anti" (red badge)
- **Conflict icon**: Warning icon if `conflicts_with` is non-empty, tooltip shows conflicting skill names

### 3.3 Experiences Page Enhancements

Current: list of experiences with action, context, outcome, valence, timestamp.

Add:
- **Chain indicator**: If `parent_id` exists, show "Parent" link that navigates to parent
- **Related count**: Badge showing `related_ids` count (e.g., "3 related")
- Click badge to expand and show related experience summaries

### 3.4 Frontend Type Updates

```typescript
// Enhanced Skill type
interface Skill {
  // ... existing fields
  skill_type: "positive" | "anti";
  status: "active" | "superseded";
  conflicts_with: string[];
  reinforcement_count: number;
  last_reinforced_at: string | null;
}

// Enhanced Experience type
interface Experience {
  // ... existing fields
  parent_id: string | null;
  related_ids: string[];
}

// New scheduler status
interface SchedulerStatus {
  enabled: boolean;
  next_decay: string | null;
  next_conflict_check: string | null;
  experience_count_since_last_crystallize: number;
  crystallize_threshold: number;
}
```

The `useApi` hook is updated to accept an optional `project` string parameter, appended as `?project=<name>` to all requests.

## Config Schema (Complete)

The full merged `config.json` with v0.3 additions. Missing keys fall back to defaults — existing v0.2 configs work without modification.

```json
{
  "storage_path": "~/.engram-ai/data",
  "llm": {"provider": "claude", "model": "claude-sonnet-4-20250514"},
  "crystallize": {"min_experiences": 3, "min_confidence": 0.7},
  "evolve": {"default_config_path": "./CLAUDE.md", "strategy": "append"},
  "default_project": "default",
  "scheduler": {
    "decay_interval_hours": 6,
    "conflict_interval_hours": 12,
    "crystallize_threshold": 10,
    "enabled": true
  }
}
```

Config loading uses `config.get("scheduler", {})` with `SchedulerConfig` defaults, so v0.2 configs without the `scheduler` or `default_project` keys work correctly.

## Dependency Order

```
1. Multi-Project (no dependencies — foundation layer)
     ↓
2. Scheduler (depends on multi-project: uses ProjectManager.get_all_forges())
     ↓
3. Dashboard Enhancement (depends on both — displays project data + scheduler status)
```

## Testing Strategy

- **ProjectManager**: Unit tests with tmp_path — project creation, listing, deletion, backward compatibility detection, invalid name rejection
- **Forge pending path**: Test that pending.jsonl is created inside storage directory, not parent
- **Scheduler**: Unit tests with mocked ProjectManager — timer tick logic, per-project counter, start/stop lifecycle, error handling (fail-open on individual project failures)
- **Dashboard API**: Extend existing test_api.py — project parameter routing, graph edge type construction, scheduler status/toggle endpoints
- **Frontend**: Manual verification (no frontend test framework currently in place)
